# Platform Update - May 15, 2026

## Scope

This note captures the repository changes completed on May 15, 2026 across product workflow, frontend architecture, runtime hardening, deployment reliability, and documentation.

## Executive Summary

The platform moved through three classes of change on May 15, 2026:

- workflow fixes in the inbox and tenant operations layer
- structural frontend work that modularized the inbox and introduced shared UI primitives
- production hardening after real runtime incidents involving malformed conversation payloads and stale frontend deployments

The result is a console that is more maintainable, more resilient to bad provider data, and closer to a formal operations product than a single-page chat implementation.

## Product and Workflow Changes

### Inbox and Operator UX

Delivered changes:

- chat messages now expose explicit copy actions
- customer profile data can be copied directly from the side panel, including phone, name, and a full customer snapshot
- the history banner and operational event copy were made more legible for day-to-day use
- the composer was simplified, removing the secondary schedule button from the primary row
- the send/record action now uses a microphone icon instead of a text label when no text or attachments are present
- image preview behavior now acts more like a full-screen lightbox, with wheel zoom and drag when zoomed

Navigation changes:

- the desktop header now supports a secondary "Mais" menu for smaller monitors
- this prevents low-priority modules from disappearing off-screen

### AI Summary Behavior

Summary behavior was narrowed to recent history:

- manual summaries now use recent messages only
- transfer summaries also use a recent-window slice rather than the full historical ticket archive
- empty recent windows return a direct "no recent messages in the last 24 hours" response

This reduces summary noise on long-lived tickets and better matches shift-handoff use cases.

### Business Hours

Business-hours evaluation was corrected in `backend/src/services/businessHourService.js`:

- default timezone is now `America/Sao_Paulo`
- inactive days are treated as closed
- decision logging was added for operational traceability

This change addressed false "outside business hours" notices during normal working periods.

### User Administration

User deletion was added to the admin workflow:

- attendants can now be removed from the console
- the deletion flow clears team membership and detaches relational ownership where safe
- deletion is blocked when the user has sent internal messages, in which case archiving is recommended instead

## Frontend Architecture Changes

### Shared UI Layer

The frontend now has a reusable UI layer under `frontend/src/components/ui/`:

- `PageHeader.jsx`
- `ActionButton.jsx`
- `SurfaceCard.jsx`
- `EmptyState.jsx`
- `ModalShell.jsx`

These primitives were applied across admin, CRM, campaign, knowledge, and superadmin pages to reduce visual drift and duplicated inline styling.

### Inbox Decomposition

The inbox is no longer implemented as a single large page file.

Current inbox composition:

- `frontend/src/pages/Inbox.jsx`
- `frontend/src/pages/inbox/components.jsx`
- `frontend/src/pages/inbox/helpers.jsx`
- `frontend/src/pages/inbox/hooks.js`

Key outcomes:

- ticket and message state moved into dedicated hooks
- rendering responsibilities moved into dedicated components
- shared helper logic was extracted from the route shell
- the page now behaves more like an orchestrator than a monolith

### Route Loading and Render Recovery

The frontend entrypoint in `frontend/src/main.jsx` now includes:

- route-level lazy loading via `React.lazy`
- `Suspense` fallback for route transitions
- an application-level error boundary

This was introduced together with inbox hardening to stop render failures from collapsing the entire app into an opaque black screen.

## Runtime Hardening

### Backend Media Retry Fix

`backend/src/services/scheduleProcessor.js` was corrected after a production crash caused by an invalid Prisma filter:

- invalid `notIn: [null, "text"]` logic was removed
- media retry now uses a valid `not: 'text'` plus `NOT: { mediaType: null }` filter
- stale pending media is marked failed, and unresolved provider media continues through the retry path

### Inbox Render Hardening

Multiple inbox fixes were applied after production regressions were observed while opening certain conversations.

Hardening themes:

- guard missing or malformed message bodies
- guard quoted-message and transcription fields
- guard missing file names and inconsistent media URLs
- guard invalid tags and absent instance names
- isolate conversation sections so a broken history payload does not take down the whole page
- expose render-error details long enough to identify the final root cause

The final identified regression was a missing `Empty` import in `frontend/src/pages/inbox/components.jsx`, which was then restored.

## Deployment and Delivery Changes

Frontend deployment behavior was tightened for EasyPanel/Nixpacks environments:

- `frontend/nixpacks.toml` now defines an explicit build phase
- `frontend/package.json` now includes a production `start` script that serves the compiled `dist`
- this was introduced after the platform repeatedly served stale frontend bundles even after redeploy

Operational lesson:

- for this repository, "deploy succeeded" was not enough validation on May 15, 2026
- the served asset hashes had to be checked against the current local build to confirm the active version
